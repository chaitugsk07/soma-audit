Architecture

Two modes. One chain.

Pick the deployment that fits your trust boundary. Both produce the same per-tenant HMAC-SHA256 chain.

Embedded

soma-audit-pg

LocalSink writes audit events directly into the host service's own Postgres. The audit row commits in the same transaction as the business write—no separate round trip, no gap between your data and the audit record. Crash between the two is architecturally impossible.

Central

soma-audit-client + server

A durable local outbox in the host's Postgres relays events to a central soma-audit-server that aggregates them across services and exposes an admin portal with chain verification, event filters, seals, and key management.

The chain is the same in both modes. HMAC-SHA256 per tenant, keys derived via HKDF-SHA256 from a 32-byte master secret so one tenant's compromise doesn't touch others. Ed25519 periodic seals checkpoint the chain head for external verifiability.
Guarantees

What you actually get.

No hand-wavy promises. These are structural guarantees that come from how the system is built.

⚡

Atomic with your write

record_in_tx commits the audit row inside your business transaction. Crash between the two is impossible — they succeed or fail together.

🔗

Tamper-evident hash chain

HMAC-SHA256 chain per tenant. Any edit, deletion, or reordering breaks the chain and is caught by verify_chain. No silent corruption.

🔒

Append-only at the database

Postgres triggers block UPDATE and DELETE on soma_audit.fct_audit_events. Even a compromised app credential can't alter the record.

🗝

Per-tenant key isolation

Keys derived from one 32-byte master secret via HKDF-SHA256. One tenant's compromise doesn't touch others. Keys never leave the server process.

✍

Ed25519 seals

Periodic signatures checkpoint the chain head for external verifiability. An auditor can verify the chain without database access.

🐘

Plain Postgres 16

No external queue, no separate process for embedded mode. Standard sqlx connection pool, no exotic Postgres features that reduce portability.

📦

Drops into any Rust app

axum / sqlx / tokio stack. install() sets up the tables via soma-schema so you don't hand-write audit SQL or manage migrations yourself.

🖥

Admin portal

Central server ships an embedded dashboard: events, filters, chain verify, seals, keys. Self-hosted, no external services.

Zero-boilerplate setup

How it sets up your database.

install(&pool) runs soma-audit's bundled migrations through soma-schema (the migration runner) and creates the soma_audit schema, the fct_audit_events table, append-only triggers, and row-level-security—coexisting with your app's own tables via a dedicated advisory lock key. You never write audit SQL by hand.

What install() creates
—-- schema
—CREATE SCHEMA IF NOT EXISTS soma_audit;
— 
—-- immutable event log
—CREATE TABLE soma_audit.fct_audit_events (
— seq BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
— tenant_id TEXT NOT NULL,
— entry_hash TEXT NOT NULL, -- HMAC-SHA256
— prev_hash TEXT NOT NULL, -- back-links the chain
— payload JSONB NOT NULL
—);
— 
—-- append-only enforcement
—CREATE TRIGGER no_mutation BEFORE UPDATE OR DELETE ON
— soma_audit.fct_audit_events FOR EACH ROW
— EXECUTE FUNCTION soma_audit.deny_mutation();
The crates

Four crates, one audit chain.

Take only what you need. Most apps add soma-audit-pg and call it done.

soma-audit-core
Pure zero-IO chain + crypto. The HMAC-SHA256 chain logic and HKDF key derivation. No database dependency.
soma-audit-pg
The embeddable LocalSink + schema installer. This is the one you add to your app. Handles install() and record_in_tx.
soma-audit-client
Remote sink: durable outbox in the host's Postgres + relay to the central server. For multi-service aggregation.
soma-audit-server
Central service: ingest, query, verify, seals, and keys API + embedded admin portal. Self-hosted, Postgres-backed.
examples/notes-app

Run the demo in 3 commands.

A runnable notes-app shows the full round-trip: create a note, read back a verified audit trail. No Docker orchestration needed—just a Postgres 16 instance.

1
Start the app
$ cd examples/notes-app $ DATABASE_URL=postgres://localhost/mydb cargo run

Connects to Postgres, runs install(), and starts the axum server on port 3000.

2
Create a note (audited automatically)
$ curl -X POST localhost:3000/notes \ -H 'Content-Type: application/json' \ -d '{"title":"hello","body":"world"}' {"id":"a3f9...","title":"hello","created_at":"2024-..."}

The note row and its audit event commit in the same transaction.

3
Verify the chain
$ curl localhost:3000/audit/verify {"ok":true,"verified":1,"tenant":"notes-app"}

verify_chain walks every record for this tenant, recomputes the HMAC, and confirms the chain hasn't been tampered with.
